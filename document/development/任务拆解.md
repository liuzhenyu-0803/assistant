# 任务拆解

本文档将 Assistant 项目拆解为可独立执行的开发任务。每个任务有明确的目标、前置依赖、预期产出和验收标准。

**拆分原则：**
- 先主链路，后增强项
- 先后端基础设施，后业务逻辑，最后前端
- 每个任务目标单一，产出可验证
- 同层任务尽量不跨层

**参考文档优先级：** 协议.md > 数据结构.md > 技术架构.md > 目录结构.md > PRD.md

---

## 阶段一：项目脚手架与共享类型

### T01 — 初始化 Monorepo 项目结构

**目标：** 搭建 pnpm workspace monorepo 骨架，让三个包（shared、server、client）可以互相引用。

**前置依赖：** 无

**预期产出：**
- 根目录 `package.json`（定义 workspace）
- `pnpm-workspace.yaml`
- `tsconfig.base.json`（共享 TypeScript 基础配置）
- `.gitignore`（含 `node_modules/`、`dist/`、`data/`）
- `packages/shared/package.json` + `tsconfig.json` + `src/index.ts`（空导出）
- `server/package.json` + `tsconfig.json` + `src/index.ts`（空 Express 启动）
- `client/package.json` + `tsconfig.json` + `vite.config.ts` + `index.html` + `src/main.tsx`（空 React 入口）
- 空目录：`skills/`、`data/`、`document/`

**验收标准：**
- `pnpm install` 成功
- `packages/shared` 可被 `server` 和 `client` 引用
- `server` 可通过 `tsx` 启动并监听端口
- `client` 可通过 `vite dev` 启动并显示空白页
- TypeScript 编译无错误

**不在本任务内：**
- 不写业务代码
- 不安装业务依赖（MCP SDK、markdown 渲染等留到使用时安装）

### T02 — 定义共享类型包

**目标：** 在 `packages/shared` 中定义全部 TypeScript 类型和工具函数，供前后端共享。

**前置依赖：** T01

**预期产出：**
- `src/types/conversation.ts` — Conversation, Fork, Summary
- `src/types/message.ts` — Message, MessageContent, MessageStatus, TextContent, AttachmentContent, ToolCallContent, SubAgentContent
- `src/types/attachment.ts` — AttachmentRef, AttachmentStatus
- `src/types/tool-call.ts` — ToolCallRecord
- `src/types/sub-agent.ts` — SubAgentRecord
- `src/types/settings.ts` — Settings
- `src/types/mcp.ts` — MCPServerConfig
- `src/types/skill.ts` — SkillMeta
- `src/types/api.ts` — API 请求/响应类型、SSE 事件类型
- `src/types/error.ts` — ErrorResponse, ErrorCode
- `src/utils/id.ts` — ID 生成函数（nanoid + 前缀）
- `src/utils/title.ts` — 会话标题生成逻辑
- `src/utils/token-estimate.ts` — 粗略 token 估算（字符数 / 3）
- `src/index.ts` — 统一导出

**验收标准：**
- 类型定义与数据结构文档完全一致
- API 类型与协议文档完全一致
- `api.ts` 覆盖协议文档第 5 节全部 SSE 事件类型（run-start、done、error、text-delta、tool-call-start/end/error、sub-agent-start/delta/end/error、summary-start/end/error、ping）
- `error.ts` 覆盖协议文档 1.4 节全部标准错误码（INVALID_REQUEST、NOT_FOUND、CONFLICT、RUN_ACTIVE、CONFIG_INCOMPLETE、INTERNAL_ERROR）
- ID 生成函数输出格式符合规范（前缀 + nanoid(12)）
- 标题生成逻辑符合 PRD 规则（30 视觉宽度，全角计 2）
- Token 估算函数返回 `Math.ceil(charCount / 3)`
- TypeScript 编译无错误
- server 和 client 均可正常导入

**不在本任务内：**
- 不写后端或前端业务代码

## 阶段二：后端存储层

### T03 — 实现存储基础设施

**目标：** 实现 JSON 原子读写、路径常量和日志工具。

**前置依赖：** T02

**预期产出：**
- `server/src/storage/paths.ts` — 数据目录路径常量（data/、conversations/、settings.json、mcp-servers.json）
- `server/src/storage/json-store.ts` — 原子 JSON 读写（写入 tmp → fsync → bak → rename；读取时主文件失败自动回退 .bak）
- `server/src/utils/logger.ts` — 简单日志工具（console 封装，带时间戳和级别标签）
- `server/src/utils/errors.ts` — 业务错误类定义（AppError，含 code、message、statusCode）

**验收标准：**
- `jsonStore.write(path, data)` 写入后 path 存在且内容正确，同时生成 .bak
- `jsonStore.read(path)` 主文件正常时返回数据
- 写入过程包含 fsync 确保数据落盘（写入 tmp → fsync → bak → rename）
- 主文件损坏时（手动破坏）自动从 .bak 恢复
- .bak 也损坏时抛出可识别的错误
- 启动时自动创建 `data/` 和 `data/conversations/` 目录（如不存在）
- AppError 可携带 HTTP 状态码和错误码

**不在本任务内：**
- 不写会话存储逻辑
- 不写路由

---

### T04 — 实现会话存储

**目标：** 实现会话索引和单会话文件的 CRUD 操作。

**前置依赖：** T03

**预期产出：**
- `server/src/storage/conversation-store.ts`
  - `listConversations()` — 读取 index.json 返回会话摘要列表
  - `getConversation(id)` — 读取单个会话完整数据
  - `saveConversation(conversation)` — 原子写入会话文件并同步更新索引
  - `deleteConversation(id)` — 删除会话文件、附件目录和索引条目
  - `deleteAllConversations()` — 清空全部会话
  - `conversationExists(id)` — 检查会话是否存在

**验收标准：**
- 新建会话后 index.json 和 conversation.json 均正确写入
- 列表接口返回按 updatedAt 降序排列的摘要
- 索引文件（index.json）仅存储摘要字段（id、title、createdAt、updatedAt、lastMessageAt、messageCount），不含 revision 和 fork；会话列表 API 中的 revision 和 fork 由 Service 层从完整会话文件中补充
- 删除会话后文件和索引条目均被移除
- 清空后 index.json 重置为空列表
- 会话不存在时返回 null 而非抛异常

**不在本任务内：**
- 不写 HTTP 路由
- 不写分叉逻辑（T11）

## 阶段三：后端核心 Service

### T05 — 实现设置 Service 与路由

**目标：** 实现设置的读写服务和 HTTP 路由。

**前置依赖：** T03

**预期产出：**
- `server/src/services/settings-service.ts`
  - `getSettings()` — 读取 settings.json，不存在时返回默认值
  - `updateSettings(settings)` — 写入 settings.json
  - `getMaskedSettings()` — 返回脱敏后的设置（apiKey 仅前3后4位）
  - `validateSettings()` — 检查 baseURL、apiKey、model 是否完整
- `server/src/routes/settings.ts`
  - `GET /api/settings` → 返回脱敏设置
  - `PUT /api/settings` → 更新设置

**验收标准：**
- GET 返回脱敏的 apiKey
- PUT 保存后再 GET 能获取到新值
- apiKey 为空或 baseURL 为空时 validateSettings 返回错误信息
- 响应格式符合协议文档（`{ data: { ... } }`）

**不在本任务内：**
- 不对接模型
- 不做 MCP 配置（T12）

### T06 — 实现会话 Service 与路由

**目标：** 实现会话的 CRUD 服务和 HTTP 路由。

**前置依赖：** T02, T04

**预期产出：**
- `server/src/services/conversation-service.ts`
  - `listConversations()` — 列表
  - `createConversation()` — 新建空会话
  - `getConversation(id)` — 获取详情（含消息）
  - `deleteConversation(id)` — 删除（含活动 Run 终止）
  - `clearAllConversations()` — 清空全部（含终止所有活动 Run，10 秒超时强制）
- `server/src/routes/conversations.ts`
  - `GET /api/conversations`
  - `POST /api/conversations`
  - `GET /api/conversations/:id`
  - `DELETE /api/conversations/:id`
  - `DELETE /api/conversations`
- `server/src/app.ts` — Express 应用配置，挂载路由

**验收标准：**
- 新建会话返回 201，含完整会话对象（revision=1, title=""）
- 列表按 updatedAt 降序返回
- 详情返回含 messages 数组的完整会话
- 删除成功返回 204
- 不存在的会话返回 404 + `NOT_FOUND` 错误码
- 清空返回 204
- 响应格式符合协议文档

**不在本任务内：**
- 不写消息发送和 Run 逻辑
- 不写分叉 API

### T07 — 实现附件 Service 与路由

**目标：** 实现附件上传、暂存、删除和文件访问。

**前置依赖：** T06

**预期产出：**
- `server/src/services/attachment-service.ts`
  - `uploadAttachment(conversationId, file)` — 保存文件到会话附件目录，返回 AttachmentRef（status=staged）
  - `deleteAttachment(conversationId, attachmentId)` — 删除 staged 状态的附件文件
  - `bindAttachments(conversationId, attachmentIds)` — 将附件状态更新为 attached
  - `getAttachmentPath(conversationId, filename)` — 返回附件文件的磁盘路径
  - `cleanOrphanAttachments(conversationId)` — 清理未绑定消息的暂存附件
- `server/src/routes/attachments.ts`（含 multer 中间件配置）
  - `POST /api/conversations/:id/attachments` — 上传
  - `DELETE /api/conversations/:id/attachments/:attachmentId` — 删除暂存附件
  - `GET /api/attachments/:conversationId/:filename` — 获取文件

**验收标准：**
- 上传文件后返回 201，含完整 AttachmentRef（status=staged）
- 上传的文件存储在 `data/conversations/{id}/attachments/` 目录下
- 文件名使用生成的文件名（非原始文件名），保留扩展名
- 删除 staged 附件返回 204，文件从磁盘移除
- 删除已 attached 附件返回错误
- 获取附件文件返回正确的 Content-Type
- 超过 100MB 的文件返回 413

**不在本任务内：**
- 不写启动时孤儿清理（T14）
- 不写运行时断连清理（T14）

### T09 — 实现 SSE 工具与 Run Service

**目标：** 实现 SSE 响应辅助工具和 Run 生命周期管理。

**前置依赖：** T06

**预期产出：**
- `server/src/utils/sse.ts`
  - `sendSSEEvent(res, event, data)` — 格式化并发送 SSE 事件
  - `setupSSEResponse(res)` — 设置 SSE 响应头
  - `startPingInterval(res)` — 启动 15 秒心跳定时器，返回清理函数
- `server/src/services/run-service.ts`
  - 内存 Map 维护活跃 Run
  - `startRun(conversationId, messageId)` — 创建并注册 Run
  - `completeRun(runId, status)` — 标记完成并从 Map 移除
  - `getActiveRun(conversationId)` — 获取会话的活跃 Run
  - `hasActiveRun(conversationId)` — 检查是否有活跃 Run
  - `stopRun(conversationId)` — 通过 AbortController 取消 Run
  - `stopAllRuns()` — 终止全部活跃 Run（含超时强制）

**验收标准：**
- SSE 事件格式正确（`event: xxx\ndata: {...}\n\n`）
- 响应头包含 `Content-Type: text/event-stream`、`Cache-Control: no-cache`、`Connection: keep-alive`
- 心跳每 15 秒发送一次 ping 事件
- Run 结束或连接关闭后心跳定时器被正确清理（无泄漏）
- 同一会话不能同时存在两个活跃 Run
- stopRun 触发 AbortController.abort()
- stopAllRuns 在 10 秒内未完成时强制终止

**不在本任务内：**
- 不写主代理逻辑
- 不写消息路由

### T10 — 实现主代理与消息路由

**目标：** 实现主代理 agentic loop 和发送消息的完整流程（不含 MCP 工具调用和子代理）。

**前置依赖：** T07, T08, T09

**预期产出：**
- `server/src/agent/system-prompt.ts` — 系统提示词构建（基础版，不含 Skill 和 MCP 工具描述）
- `server/src/agent/main-agent.ts`
  - `runMainAgent(conversation, userMessage, sseWriter, abortSignal)` — 主代理执行循环
  - 构建模型输入（系统提示 + 消息历史）
  - 调用模型，流式接收响应
  - 纯文本回复 → 通过 sseWriter 发送 text-delta → 结束循环
  - tool_calls 回复 → 暂时跳过工具执行，记录工具调用信息 → 继续循环（后续任务补充真实工具调用）
  - 支持 AbortSignal 中断
- `server/src/routes/messages.ts`
  - `POST /api/conversations/:id/messages` — 接收消息，校验 revision，启动 Run，返回 SSE 流
  - `POST /api/conversations/:id/stop` — 停止当前 Run
- 更新 `server/src/app.ts` — 挂载 messages 路由
- 完善 `server/src/services/conversation-service.ts`
  - `addUserMessage(id, content, revision)` — 添加用户消息（含 revision 冲突检测、附件绑定、attachment 类型 content 展开为完整 AttachmentRef）
  - `addAssistantMessage(id, message)` — 保存助手回复
  - 会话标题自动生成（首条消息时）

**验收标准：**
- 发送消息后收到 SSE 流，依次包含 run-start → text-delta(s) → done 事件
- revision 不匹配时返回 409 CONFLICT
- 会话已有活动 Run 时返回 409 RUN_ACTIVE
- 配置不完整时返回 422 CONFIG_INCOMPLETE
- 停止 Run 后收到 done 事件（status=interrupted），已输出内容保留
- 首条消息发送后会话标题自动生成
- 助手回复在 Run 完成时持久化
- 用户消息在接收时立即持久化
- 前端断开连接（req.close）后 Run 被标记为中断

**不在本任务内：**
- 不实现真实的 MCP 工具调用（T12）
- 不实现子代理委派（T13）
- 不实现上下文摘要（T15）
