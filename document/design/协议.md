# 协议

本文档定义 Assistant 前后端之间的全部 HTTP API 接口、SSE 事件格式和错误规范。

## 1. 通用约定

### 1.1 请求格式

- Content-Type: `application/json`（文件上传除外，使用 `multipart/form-data`）
- 字符编码：UTF-8

### 1.2 成功响应格式

```json
{
  "data": { ... }
}
```

列表接口：

```json
{
  "data": [ ... ]
}
```

无返回体的操作返回 HTTP `204 No Content`。

### 1.3 错误响应格式

```json
{
  "error": {
    "code": "ERROR_CODE",
    "message": "人类可读的错误描述"
  }
}
```

### 1.4 标准错误码

| HTTP 状态码 | code | 含义 |
|------------|------|------|
| 400 | `INVALID_REQUEST` | 请求参数不合法 |
| 404 | `NOT_FOUND` | 资源不存在 |
| 409 | `CONFLICT` | 会话版本冲突或已有活动 Run |
| 409 | `RUN_ACTIVE` | 会话已有进行中的 Run |
| 422 | `CONFIG_INCOMPLETE` | 模型配置不完整 |
| 500 | `INTERNAL_ERROR` | 服务端内部错误 |

```
DELETE /api/conversations/:id
```

**响应 204：** 无返回体

### 2.5 清空所有会话

```
DELETE /api/conversations
```

服务端行为：
1. 终止所有活动 Run（10 秒超时后强制中止）
2. 删除全部会话文件及附件目录
3. 重置会话索引

**响应 204：** 无返回体

## 3. 消息与 Run API

### 3.1 发送消息

```
POST /api/conversations/:id/messages
```

**请求体：**

```json
{
  "revision": 6,
  "content": [
    { "type": "text", "text": "帮我分析这张图片" },
    { "type": "attachment", "attachmentId": "att_001" },
    { "type": "attachment", "attachmentId": "att_002" }
  ]
}
```

**冲突检测：** 后端比较请求中的 `revision` 与服务端当前值：
- 一致 → 接受请求，revision +1，开始 Run
- 不一致 → 返回 409

**附件绑定：** 请求中的 `attachmentId` 引用先前上传的暂存附件。后端接收消息后，将暂存附件状态从 `staged` 更新为 `attached`，并在持久化的消息内容中展开为完整的 `AttachmentRef`（含 filename、mimeType、size 等字段，格式见数据结构文档）。

**重试失败 Run：** 若前一轮 Run 以 `failed` 状态结束，前端使用相同内容重新调用本接口即可重试，无需额外 API。前端应使用最新的 `revision`（从上次失败响应的 `done` 或 `error` 事件中获取）。

**响应 409（版本冲突）：**

```json
{
  "error": {
    "code": "CONFLICT",
    "message": "会话版本冲突，当前版本为 7"
  }
}
```

**响应 409（已有活动 Run）：**

```json
{
  "error": {
    "code": "RUN_ACTIVE",
    "message": "该会话正在生成中"
  }
}
```

**响应 200（SSE 流）：**

```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

SSE 事件流格式见第 5 节。

### 3.2 停止当前 Run

```
POST /api/conversations/:id/stop
```

**响应 200：**

```json
{
  "data": {
    "stopped": true
  }
}
```

**响应 404：** 无活动 Run

## 4. 附件 API

### 4.1 上传附件

```
POST /api/conversations/:id/attachments
Content-Type: multipart/form-data
```

**表单字段：**
- `file`: 文件二进制数据

**大小限制：** 建议单个文件不超过 100 MB。超出时服务端返回 `413 Payload Too Large`。前端在选择文件时预先检查大小并显示警告，但不强制阻止上传。

**响应 201：**

```json
{
  "data": {
    "id": "att_001",
    "filename": "a1b2c3d4.png",
    "originalName": "screenshot.png",
    "mimeType": "image/png",
    "size": 204800,
    "status": "staged",
    "createdAt": "2025-01-01T00:00:00.000Z"
  }
}
```

### 4.2 删除未绑定附件

```
DELETE /api/conversations/:id/attachments/:attachmentId
```

仅允许删除 `status: "staged"` 的附件，已绑定消息的附件不可删除。

**响应 204：** 无返回体

### 4.3 获取附件文件

```
GET /api/attachments/:conversationId/:filename
```

**响应 200：** 文件二进制流，附带正确的 `Content-Type` 和 `Content-Disposition`。

## 5. SSE 事件协议

### 5.1 事件格式

每个 SSE 事件由 `event` 和 `data` 两行组成：

```
event: <事件类型>
data: <JSON 数据>

```

事件之间以空行分隔。

### 5.2 Run 生命周期事件

#### run-start

Run 开始，携带 Run ID 和助手消息 ID。

```
event: run-start
data: {"runId":"run_001","messageId":"msg_002"}
```

#### done

Run 正常完成或被用户中断。

```
event: done
data: {"runId":"run_001","status":"completed","revision":7}
```

`status` 取值：`completed`（正常完成）| `interrupted`（用户中断，部分内容已保留）。

#### error

Run 执行出错。

```
event: error
data: {"runId":"run_001","code":"MODEL_ERROR","message":"模型请求超时","revision":7}
```

### 5.3 文本与思考输出事件

#### text-delta

模型输出的最终回答增量文本片段。

```
event: text-delta
data: {"delta":"你好"}
```

#### reasoning-delta

模型输出的思考过程增量文本片段。该事件用于前端独立展示 thinking / reasoning，不与最终回答正文混排。

```
event: reasoning-delta
data: {"delta":"先分析用户的问题..."}
```

#### reasoning-end

模型思考过程结束。前端可用它将“思考进行中”状态切换为“思考过程已完成”。

```
event: reasoning-end
data: {"status":"completed"}
```

### 5.4 工具调用事件

#### tool-call-start

开始执行工具调用。

```
event: tool-call-start
data: {"toolCallId":"tc_001","toolName":"read_file","arguments":{"path":"/tmp/data.csv"}}
```

#### tool-call-end

工具调用完成。

```
event: tool-call-end
data: {"toolCallId":"tc_001","result":"文件内容...","status":"success"}
```

#### tool-call-error

工具调用失败。

```
event: tool-call-error
data: {"toolCallId":"tc_001","error":"MCP Server 不可达","status":"failed"}
```

### 5.5 子代理事件

#### sub-agent-start

子代理任务开始。

```
event: sub-agent-start
data: {"subAgentId":"sa_001","task":"分析CSV文件中的数据趋势"}
```

#### sub-agent-delta

子代理执行过程中的进度更新。前端可选择性展示为子代理执行状态文本，帮助用户了解子代理当前正在做什么。

```
event: sub-agent-delta
data: {"subAgentId":"sa_001","delta":"正在读取文件..."}
```

#### sub-agent-end

子代理任务完成。

```
event: sub-agent-end
data: {"subAgentId":"sa_001","summary":"数据显示Q1增长15%...","status":"success","detail":"完整分析过程..."}
```

#### sub-agent-error

子代理任务失败。

```
event: sub-agent-error
data: {"subAgentId":"sa_001","error":"执行超时","status":"failed"}
```

### 5.6 摘要事件

#### summary-start

开始生成上下文摘要。

```
event: summary-start
data: {"upToMessageId":"msg_010"}
```

#### summary-end

摘要生成完成。

```
event: summary-end
data: {"upToMessageId":"msg_010","status":"success"}
```

#### summary-error

摘要生成失败，回退为截断。

```
event: summary-error
data: {"upToMessageId":"msg_010","message":"摘要生成失败，已截断较早消息"}
```

### 5.7 心跳事件

#### ping

服务端在 SSE 流空闲期间（如等待工具调用或子代理执行）每 **15 秒** 发送一次心跳，防止浏览器或中间代理因 idle timeout 断开连接。

```
event: ping
data: {}
```

## 9. 前端分叉流程协议

当前端收到 `409 CONFLICT` 响应时，执行以下流程：

1. 调用 `GET /api/conversations/:id` 获取当前最新会话数据
2. 调用 `POST /api/conversations` 创建新会话
3. 新会话标题 = 来源标题 + "（分支）"
4. 将来源会话截至冲突点的消息复制到新会话（通过后续的消息发送完成）
5. 前端跳转到新会话，保留用户草稿
6. Toast 提示"会话已被其他标签页更新，已自动创建分支会话"

或者，提供一个专用的分叉 API：

### 9.1 分叉会话

```
POST /api/conversations/:id/fork
```

**请求体：**

```json
{
  "upToMessageId": "msg_005",
  "revision": 6
}
```

**响应 201：**

```json
{
  "data": {
    "id": "conv_new789",
    "title": "如何实现一个Web助手（分支）",
    "fork": {
      "sourceConversationId": "conv_abc123",
      "sourceMessageId": "msg_005",
      "sourceRevision": 6
    },
    "revision": 1,
    "messages": [ ... ]
  }
}
```

服务端行为：
1. 复制来源会话从起点到 `upToMessageId` 的消息历史
2. 为新会话中的每条消息生成新 `messageId`
3. 异步复制附件文件到新会话目录
4. 未完成拷贝的附件标记为 `status: "copying"`
