# 技术架构

## 1. 技术选型

### 1.1 前端

| 类别 | 选型 | 说明 |
|------|------|------|
| 框架 | React 18 | 成熟生态，Hooks 模型适合状态驱动 UI |
| 语言 | TypeScript | 与后端共享类型 |
| 构建 | Vite | 开发体验好，HMR 快 |
| 状态管理 | Zustand | 轻量、无 boilerplate，适合中小规模应用 |
| Markdown 渲染 | react-markdown + remark-gfm + rehype-highlight | GFM 支持 + 代码高亮 |
| HTTP 请求 | fetch API | 原生支持 SSE 流式读取，无需额外库 |
| 样式 | CSS Modules | 简单可靠，无运行时开销 |

### 1.2 后端

| 类别 | 选型 | 说明 |
|------|------|------|
| 运行时 | Node.js 20+ | LTS，原生支持 ESM |
| 框架 | Express | 成熟稳定，中间件生态丰富 |
| 语言 | TypeScript | 与前端共享类型 |
| 开发运行 | tsx | TypeScript 直接执行，无需编译步骤 |
| 文件上传 | multer | Express 标准文件上传中间件 |
| UUID | crypto.randomUUID() | Node.js 内置，无需额外依赖 |
| 日志 | 标准输出 | 当前阶段不引入日志框架，直接 console + 简单格式化 |

### 1.3 共享

| 类别 | 选型 | 说明 |
|------|------|------|
| 包管理 | pnpm | Monorepo workspace 支持好 |
| Monorepo | pnpm workspace | 共享类型包 `packages/shared` |
| MCP SDK | @modelcontextprotocol/sdk | 官方 SDK，支持全部 transport |

## 3. 后端分层

### 3.1 层次职责

| 层 | 职责 | 规则 |
|---|------|------|
| **Routes** | HTTP 入口，参数校验，响应格式化 | 不含业务逻辑 |
| **Services** | 业务编排，Run 生命周期，冲突检测 | 可调用 Agent、Storage、MCP |
| **Agent** | 模型调用，agentic loop，子代理管理 | 通过回调/事件向 Service 报告进度 |
| **MCP** | MCP 客户端连接管理，工具发现与调用 | 封装 transport 差异 |
| **Skills** | Skill 扫描、解析、元信息管理 | 只读，启动时加载 |
| **Storage** | JSON 文件读写，原子写入，备份恢复 | 唯一的持久化出口 |

### 3.2 依赖方向

```
Routes → Services → Agent → MCP
                          → Skills
                  → Storage
```

禁止反向依赖：Storage 不依赖 Service，Agent 不依赖 Route。

## 4. 主代理 Agentic Loop

主代理在一个 Run 中执行循环，直到模型决定终止或被用户中断：

```
开始 Run
  │
  ▼
构建模型输入（会话历史 + 系统提示 + 工具定义 + Skill 指令）
  │
  ▼
┌─────────────────────────────┐
│  调用模型（流式）            │◄──────────────────┐
│  逐 token 推送 text-delta   │                   │
│  按需推送 reasoning-delta   │                   │
└──────────┬──────────────────┘                   │
           │                                      │
           ▼                                      │
     模型输出类型判断                              │
       │         │          │                     │
       ▼         ▼          ▼                     │
    纯文本   tool_calls   子代理委派              │
       │         │          │                     │
       ▼         ▼          ▼                     │
    结束Run  执行工具    提交到SubAgentPool        │
              │          等待结果                  │
              ▼          │                        │
           工具结果      ▼                        │
              │       摘要结果                    │
              └──────────┴────────────────────────┘
                    合并结果，继续下一轮模型调用
```

### 4.1 模型调用方式

通过 OpenAI 兼容的 `chat/completions` 接口，设置 `stream: true`：

- 请求体包含 `messages`、`tools`（从 MCP 工具列表转换）、`model` 等
- 流式读取 SSE 响应，解析 `delta` 中的 `content`、兼容 provider 的 `reasoning/thinking` 字段和 `tool_calls`
- 对 reasoning / thinking 仅做运行时流式展示，不写入会话持久化结构，也不回灌到后续模型上下文
- 当模型返回 `finish_reason: "tool_calls"` 时进入工具执行分支
- 当模型返回 `finish_reason: "stop"` 时结束循环

### 4.2 工具定义来源

模型请求中的 `tools` 数组来自两个来源：

1. **MCP 工具** — 从已连接的 MCP Server 获取工具列表，转换为 OpenAI function calling 格式
2. **子代理委派工具** — 注入一个虚拟工具 `delegate_to_sub_agent`，参数为任务描述和上下文摘要

### 4.3 停止生成

- 前端发送 `POST /api/conversations/:id/stop`
- 后端通过 `AbortController` 取消当前模型流式请求
- 同时取消所有活动的子代理和 MCP 工具调用
- 已输出内容保留，消息标记为 `interrupted`

## 5. 子代理执行架构

### 5.1 SubAgentPool

```typescript
// 核心结构
class SubAgentPool {
  private running: Map<string, SubAgentTask>  // 当前运行中的子代理
  private queue: SubAgentTask[]               // 等待队列
  private readonly maxConcurrent = 3
  private readonly queueTimeout = 30_000      // 排队超时 30s
  private readonly execTimeout = 120_000      // 执行超时 120s
}
```

### 5.2 执行流程

1. 主代理通过 `delegate_to_sub_agent` 工具发起委派
2. SubAgentPool 检查并发数：
   - 未满 → 立即启动子代理
   - 已满 → 入队等待，超过 30s 未获得空位 → 回退给主代理自行处理
3. 子代理独立执行 agentic loop（可调用 MCP、Skill，不可再委派）
4. 子代理完成后返回摘要结果
5. 主代理将摘要纳入上下文，继续循环

### 5.3 子代理与主代理的隔离

- 子代理拥有独立的消息历史（仅包含任务描述 + 执行过程）
- 子代理不接收主会话的完整历史
- 子代理执行结果仅以摘要形式回传
- 子代理的中间过程不写入会话持久化存储

## 6. SSE 流式通信方案

### 6.1 方案选择

采用 **POST 请求返回 SSE 流** 的方式，而非 EventSource API：

- EventSource 只支持 GET，无法携带请求体
- 使用 `fetch` + `ReadableStream` 手动解析 SSE 格式
- 响应 `Content-Type: text/event-stream`

### 6.2 前端 SSE 读取

```typescript
// 伪代码示意
const response = await fetch(url, {
  method: 'POST',
  body: JSON.stringify(payload),
  signal: abortController.signal,
})

const reader = response.body.getReader()
const decoder = new TextDecoder()

while (true) {
  const { done, value } = await reader.read()
  if (done) break
  const text = decoder.decode(value, { stream: true })
  // 解析 SSE 格式：event: xxx\ndata: {...}\n\n
  parseSSEEvents(text).forEach(handleEvent)
}
```

### 6.3 连接生命周期

- 前端发送消息时建立 SSE 连接
- 连接在 Run 完成（收到 `done` 事件）或中断时关闭
- 前端通过 `AbortController` 可主动断开
- 后端通过监听 `req.on('close')` 检测客户端断开，触发后标记当前 Run 为中断
- 服务端在 SSE 流空闲期间每 **15 秒** 发送 `ping` 心跳事件，防止浏览器或中间代理因 idle timeout 断开连接

## 7. MCP 客户端实现

### 7.1 MCPManager

统一管理所有 MCP Server 连接：

```
MCPManager
  ├── MCPConnection (stdio)
  │     └── StdioClientTransport → 本地进程
  ├── MCPConnection (streamable-http)
  │     └── StreamableHTTPClientTransport → 远程服务
  └── MCPConnection (sse)
        └── SSEClientTransport → 兼容旧版远程服务
```

### 7.2 生命周期

- 应用启动时根据配置初始化所有 MCP 连接
- 配置变更时断开旧连接、建立新连接
- 应用关闭时优雅断开所有连接
- 连接失败时标记为不可用，不阻塞其他 Server

### 7.3 工具发现与调用

- 连接建立后调用 `client.listTools()` 获取工具列表
- 工具定义转换为 OpenAI function calling 格式供模型使用
- 工具调用时根据工具名路由到对应的 MCP 连接执行

### 7.4 工具调用超时

- 单次 MCP 工具调用超时时间为 **60 秒**
- 超时后标记该工具调用为 `failed`，代理可选择跳过或使用替代方案继续回答
- 子代理内部的 MCP 调用同样受 60 秒超时约束，与子代理总执行超时（120 秒）独立判定——任一超时即触发

## 8. 存储层设计

### 8.1 原子写入

所有 JSON 文件写入遵循以下步骤：

```
1. 将数据写入临时文件 target.json.tmp
2. 确认写入完成（fsync）
3. 若已有 target.json，复制为 target.json.bak
4. 将 target.json.tmp 重命名为 target.json（原子操作）
```

### 8.2 异常恢复

读取 JSON 文件时：

```
1. 尝试读取 target.json
2. 若解析失败，尝试读取 target.json.bak
3. 若 .bak 也失败，标记为不可读
```

### 8.3 持久化时机

| 数据 | 持久化时机 |
|------|-----------|
| 用户消息 | 接收时立即写入 |
| 助手回复 | Run 完成或中断时写入 |
| 会话索引 | 会话创建/删除/更新时写入 |
| 设置 | 用户保存时写入 |
| MCP 配置 | 用户保存时写入 |
| 模型 Provider 配置 | 编辑失去焦点或关闭设置对话框时写入 |

## 9. 上下文摘要机制

### 9.1 触发条件

每次构建模型输入前，估算当前消息历史的 token 数：

- 采用近似估算：`字符数 / 3`（粗略兼容不同模型）
- 当估算 token 数超过配置的上下文窗口容量的 50% 时触发摘要

### 9.2 摘要流程

1. 将较早的消息提取出来，发送独立的摘要请求给当前模型
2. 模型返回摘要文本
3. 摘要持久化到会话文件中
4. 后续模型请求用摘要替代原始消息作为上下文前缀
5. 原始消息保留在存储和前端展示中

### 9.3 摘要失败回退

若摘要请求失败，直接截断较早消息以适配上下文窗口，并在回复中提示上下文可能不完整。

## 10. 附件生命周期

### 10.1 状态流转

```
用户选择/拖拽/粘贴文件
  │
  ▼
上传到服务端暂存 → status: staged
  │
  ├── 用户发送前删除 → 服务端立即删除文件和记录
  │
  └── 用户发送消息 → 绑定到消息 → status: attached
                                    │
                                    └── 会话分叉时 → 异步复制 → status: copying → 完成后 → attached
```

### 10.2 孤儿清理

未绑定到任何消息的 `staged` 附件可能因用户关闭页面而遗留，需要两级清理：

1. **启动时清理** — 服务端启动时扫描全部会话的附件目录，删除未被任何消息引用的暂存附件文件
2. **运行时清理** — 用户连接断开后，服务端等待 **5 分钟**（容忍页面刷新等短暂断连）后清理该会话中未绑定消息的暂存附件

### 10.3 附件与会话生命周期

- 每个会话拥有独立的附件文件副本，不与其他会话共享
- 删除会话时直接删除该会话目录下的全部附件文件
- 分叉时为新会话异步复制附件文件，复制完成前标记为 `copying`

## 11. Skill 匹配机制

### 11.1 加载时机

应用启动时扫描 `skills/` 目录，解析全部 `SKILL.md` 的 YAML front matter，将 Skill 元信息（name、description、match）加载到内存。若某个 Skill 解析失败或目录结构不合法，跳过并记录警告，不影响其他 Skill。

### 11.2 匹配与注入

- 系统提示词中包含已加载 Skill 的摘要列表（name + description + match 关键词）
- 代理（主代理或子代理）根据当前任务内容与 Skill 的 `match` 字段自主判断是否使用
- 当代理决定使用某个 Skill 时，将该 Skill 的 `SKILL.md` 正文内容注入为当前轮次的附加系统指令
- Skill 不直接替代 MCP，而是为代理提供额外知识、流程和执行约束

## 12. 会话标题生成

首条用户消息发送后立即生成会话标题，规则如下：

1. 取消息文本去除 Markdown 标记和空白字符
2. 按视觉宽度截取前 **30 个视觉宽度单位**（全角字符计 2，半角字符计 1）
3. 若首条消息无可用文本，回退为首个附件文件名
4. 若仍无法生成，使用创建时间格式化为回退标题
5. 分支会话标题在来源标题后追加"（分支）"

前端展示时超出列表宽度的标题截断并显示省略号。

## 13. 优雅关闭

服务端收到终止信号（SIGTERM / SIGINT）时，按以下顺序关闭：

```
1. 停止接受新的 HTTP 请求
2. 向所有活动 Run 发送中止信号
3. 等待活动 Run 将已产生的内容持久化为 interrupted 状态
4. 断开所有 MCP 连接
5. 释放文件句柄
6. 退出进程
```

若步骤 2-3 在合理时间内未完成，强制终止剩余 Run 后继续后续步骤。
