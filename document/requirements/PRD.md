# Assistant PRD

## 1. 文档目标

本文档用于明确 Assistant 产品的目标、用户、核心场景、产品规则、功能范围与验收标准，为后续技术方案和 AI 自动开发提供稳定边界。

## 2. 产品概述

Assistant 是一个纯 Web 的单用户 AI 助手产品，交互风格参考 Claude Desktop，不对接第三方 IM。用户通过自研前端界面向后端发送消息，后端基于 OpenAI 兼容接口完成模型调用，并具备 Skills、MCP 与通用子代理能力。

默认通过 OpenAI 兼容 `chat/completions` 发起模型请求，并使用 `SSE` 向前端返回流式结果。

产品强调以下能力：

- 纯 Web 使用体验
- 基于本地 JSON 文件与本地文件目录的轻量存储
- 会话式聊天与历史恢复
- 文件上传、拖拽、粘贴输入
- 技能包（Skills）加载能力
- MCP 工具集成能力
- 通过子代理隔离执行，降低主代理上下文污染

### 2.1 核心术语

- **Run（执行轮次）**：一次完整的请求—响应周期。从用户发送消息开始，到助手回复全部生成完毕（或被中断、失败）结束。一个 Run 可能包含模型调用、工具/MCP 调用、子代理委派等步骤，整体作为一个原子单元绑定到一条助手回复上。每个会话同一时刻最多只有一个活动 Run。

## 3. 产品目标

### 3.1 目标

- 提供一个可长期自用的 Web Assistant 基础产品。
- 让用户能像使用主流桌面 AI 客户端一样进行连续对话、上传文件、查看历史会话。
- 在不引入数据库的前提下，完成稳定的本地持久化。
- 兼容 OpenAI 风格模型接口，允许灵活切换 `baseURL`、`apiKey`、`model`。
- 支持 Skills、MCP、子代理三类能力协同工作。
- 通过通用子代理机制，减少复杂任务对主代理上下文的污染。

### 3.2 非目标

以下能力不属于当前版本：

- 第三方 IM 接入
- 多用户系统
- 登录、鉴权、权限体系
- Docker 部署
- 复杂会话管理，如重命名、归档、标签
- Skill 商店、MCP 商店、插件市场
- 可视化工作流编排系统
- 会话搜索与导出

## 4. 目标用户

### 4.1 核心用户

- 单个开发者或高频 AI 用户
- 希望自托管或本地运行一个可扩展 Assistant 的用户
- 希望兼容 OpenAI 风格接口，并能接入 Skills / MCP 的用户

### 4.2 用户特征

- 可以接受本地文件存储
- 愿意手动配置模型参数
- 更关注可扩展性和可控性，而非复杂协作能力

## 5. 核心使用场景

### 5.1 基础聊天

用户打开应用，新建会话，输入问题，与模型进行多轮对话，并可随时查看历史会话继续交流。

### 5.2 文件辅助问答

用户通过上传、拖拽或粘贴的方式添加文件，将任意浏览器可上传的文件作为附件随消息发送。图片在模型支持视觉输入时直接作为模型输入的一部分；其他文件默认不直接传递给模型，而是由代理通过 MCP 工具（包括系统内置的文件系统、命令行等 MCP）完成读取、解析、提取或转换后，再进入后续处理。

### 5.3 图片理解

用户上传或粘贴图片作为附件发送，前端以缩略图展示，后端在模型支持视觉输入时将图片作为模型输入的一部分。

### 5.4 技能驱动任务

用户提出特定类型任务时，系统可加载匹配的 Skill，为主代理或子代理提供额外知识、流程和约束。

### 5.5 工具与 MCP 调用

用户提出需要外部工具支持的任务时，主代理或子代理可自动调用 MCP 工具完成处理，并将结果回传到对话中。

### 5.6 子代理隔离执行

当主代理判断某项任务只需要结果而不需要保留完整中间过程时，可将任务委派给子代理，由子代理自主决定是否使用工具、MCP、Skill，并仅将摘要结果回传给主代理。

## 6. 功能范围

### 6.1 聊天与会话

- 新建会话
- 历史会话列表
- 进入历史会话继续聊天
- 支持多个标签页或窗口中对不同会话并行问答
- 同一会话支持多标签页查看，冲突时自动分叉为新会话
- 删除会话
- 清空所有会话
- 会话标题根据首条消息内容自动生成
- 会话消息流式输出
- 支持展示模型思考过程（reasoning / thinking）的流式内容
- Markdown 渲染
- 代码块渲染与复制
- 停止当前轮生成

### 6.2 文件与附件

- 任意类型文件选择上传
- 拖拽上传
- 编辑框粘贴文件
- 发送前附件预览
- 发送前移除单个附件
- 发送后在消息中展示附件
- 图片附件缩略图展示
- 非图片附件卡片展示
- 图片附件点击预览大图

### 6.3 模型与集成配置

- 模型 Provider 配置存储在 `data/model-providers.json` 文件中
- 支持在系统配置界面直接编辑 Provider 配置 JSON
- 编辑完成后失去焦点或点击关闭按钮自动保存
- 配置修改后立即生效，可在下拉列表中选择新增的 Provider
- 配置字段包括：`name`、`baseURL`、`apiKey`、`models`
- 前端界面可选择 Provider 和 Model
- API Key 可以存储在配置文件的 `apiKey` 字段中
- MCP Server 配置存储在 `data/mcp-servers.json` 文件中
- 系统配置界面直接展示 MCP 配置文件 JSON 编辑框，不提供单独的新增/删除/编辑表单入口
- 用户仅通过编辑配置文件完成 MCP Server 的新增、删除和修改
- MCP 配置编辑框失去焦点后自动保存，保存成功后刷新 MCP Server 列表与启用状态
- 若已配置 MCP Server，则在配置编辑框上方展示 Server 列表、transport、连接状态和启用开关
- 点击启用开关后立即通知后端更新对应配置，更新成功后刷新下方 MCP 配置文件内容
- 保存 MCP 配置后后端立即重新初始化连接，本地保存配置，应用启动后恢复配置
- 请求前配置完整性校验
- 模型能力不足提示

### 6.4 Agent 能力

- 主代理直接回复
- 主代理直接调用工具 / MCP
- 主代理委派给子代理
- 子代理自主选择工具 / MCP / Skill
- 子代理结果摘要回传
- 工具状态与子代理结果展示

### 6.5 Skills 与 MCP

- 启动时扫描应用内 Skills 目录，加载全部 Skill 元信息
- 配置 MCP Server 列表
- 支持主流 MCP transport（`stdio`、`streamable HTTP`、兼容 `HTTP+SSE`）
- 自动发现并调用 MCP 工具

### 6.6 存储

- 本地 JSON 文件保存设置
- 本地 JSON 文件保存会话与消息
- 本地目录保存附件原文件
- 应用启动恢复历史数据
- 原子写入降低文件损坏风险

## 7. 核心产品规则

### 7.1 会话规则

- 每个会话独立存储。
- 单用户可以同时在多个标签页或窗口中打开不同会话并行使用。
- 同一会话可以被多个标签页同时查看，但会话历史保持线性，不做多端协同合并。
- 同一会话同一时刻最多只有一个活动 `run`；不同会话可以各自独立生成。
- 每个会话维护一个 `revision` 字段，类型为从 `1` 起的自增整数，每次会话消息历史发生变更（新增用户消息、Run 完成或中断）时递增。前端在加载会话时获取当前 `revision`，发送消息时携带本地已知的 `revision` 向后端发起请求；后端比较该值与服务端当前 `revision`，若不一致则拒绝写入并返回冲突状态码。
- 前端收到冲突响应后，自动从当前节点分叉为新会话并保留用户草稿，无需用户手动选择。
- 分叉完成后，前端以 Toast 或横幅方式通知用户"会话已被其他标签页更新，已自动创建分支会话"。
- 分支出的新会话需要复制来源会话从起点到目标消息的线性历史，并记录来源会话、来源消息与来源修订快照，用于表达"从这里另开一条线"。
- 分支会话复制历史消息内容时需要重新生成新会话内的 `messageId`；历史消息中引用的附件文件异步拷贝一份副本到新会话的附件目录下，拷贝完成前附件标记为待就绪状态；新会话持有独立的附件文件，不与来源会话共享。
- 会话标题在首条用户消息发送后立即生成，取消息文本去除 Markdown/空白后截取前若干可见字符，上限为 `30` 个视觉宽度单位（全角字符计 2，半角字符计 1）；若首条消息无可用文本，则回退为首个附件文件名；若仍无法生成，则使用创建时间作为回退标题。前端展示时超出列表宽度的标题截断并显示省略号。
- 分支会话标题默认在来源标题后追加 `（分支）`。
- 会话列表至少展示标题、更新时间与最后一条消息时间；默认按最后更新时间降序排列。
- 多标签页会话列表同步：前端在标签页获得焦点（`visibilitychange`）时自动拉取最新会话列表；新建、删除会话等写操作完成后，当前标签页立即刷新列表。若标签页 A 删除了某会话，标签页 B 在刷新列表后该会话消失；若 B 在刷新前尝试进入已删除会话，后端返回 `404`，前端提示"该会话已被删除"并跳转到会话列表。
- 清空所有会话时，需要二次确认；确认后删除全部会话文件及其附件目录，会话索引重置为空；若任意会话中有正在进行的 `run`，需先终止全部活动 `run` 后再执行清空；若终止操作在 `10` 秒内未全部完成，系统强制中止剩余 `run` 并继续执行清空。

### 7.2 消息规则

- 消息支持文本、附件、工具状态、子代理结果等内容。
- 模型思考过程仅用于前端会话内流式展示，不写入消息持久化结构，也不回灌到后续模型上下文。
- 同一会话内消息只允许按尾部线性追加，不支持多个标签页同时改写同一历史节点。
- 每条助手回复只属于一个 `run`，且 `run` 只能绑定一个会话。
- 所有发送出去的附件都必须在消息中展示。
- Markdown 为默认文本展示格式。
- 代码块需要支持复制。
- 当会话消息累计 token 数超过模型上下文窗口容量的 `50%` 时，系统自动对较早的消息生成摘要。token 计数采用近似估算以兼容不同模型后端。摘要通过向当前配置的模型发起独立请求生成，生成后替代原始消息作为后续模型请求的上下文输入，完整消息历史仍保留在存储和前端展示中。
- 摘要结果持久化到会话文件中，服务端重启后无需重新生成。
- 若摘要请求失败，系统回退为截断较早的消息以适配上下文窗口，并在回复中附带提示说明上下文可能不完整。

### 7.3 附件规则

- 上传、拖拽、粘贴得到的文件统一作为消息附件处理。
- 附件在用户选择、拖拽或粘贴时立即上传至服务端暂存，发送消息时绑定到对应消息。
- 前端允许上传任意浏览器可选择的文件类型，不做文件类型和数量限制；单个文件大小建议上限为 `100 MB`，超出时前端显示警告提示但不强制阻止。
- 粘贴图片不作为正文内联内容，而是作为附件处理。
- 图片附件以缩略图展示。
- 非图片附件以附件卡片展示。
- 发送前允许删除待发送附件。
- 发送前删除待发送附件时，前端需要通知服务端立即删除尚未被任何消息引用的附件文件与索引记录。
- 图片附件在模型支持视觉输入时可直接作为模型输入的一部分。
- 非图片附件默认不直接作为模型原生输入，而是优先通过 MCP 工具完成解析、提取或格式转换后再参与后续推理。
- 若当前环境没有可用的 MCP 解析能力，系统需要明确提示该附件暂不可直接处理，但仍保留附件展示、打开和下载能力。
- 发送后图片附件支持点击预览大图；非图片附件以卡片形式展示，支持下载原文件。
- JSON 中仅记录附件元信息与文件引用，原文件本体保存在本地目录。
- 附件跟随会话生命周期管理：每个会话拥有独立的附件文件副本，删除会话时直接删除该会话目录下的全部附件文件。
- 暂存附件的孤儿清理：服务端启动时扫描全部会话的附件目录，清理未被任何消息引用的暂存附件文件；运行期间，若用户关闭页面或连接断开，服务端等待 `5` 分钟后清理该会话中未绑定消息的暂存附件，以容忍页面刷新等短暂断连。

### 7.4 停止生成规则

- 用户点击停止后，系统默认尝试取消当前轮模型流式输出。
- 系统默认尝试取消当前轮关联的工具、MCP、子代理任务。
- 若底层任务不支持取消，则后续结果不再回写到当前消息。
- 停止操作只作用于当前轮任务，不影响历史消息。
- 已输出的部分内容保留在消息中，该消息标记为 `interrupted` 状态，前端以视觉标记提示该回复未完成。

### 7.5 主代理与子代理规则

- 主代理负责决定：直接回答、直接调用工具/MCP、或委派子代理。
- 子代理负责决定：是否调用工具、MCP、Skill，以及调用顺序。
- 子代理的唯一核心目标是降低主代理上下文污染。
- 子代理默认只向主代理返回摘要、结论和必要产物。
- 子代理执行详情可以作为展开信息展示，但默认不进入主回复正文。
- 子代理的工具、MCP、Skill 可见范围与主代理一致，不做额外限制或子集筛选。
- 子代理不允许递归委派（委派深度固定为 1 层：主代理 → 子代理）。
- 单个 `run` 中最多同时存在 `3` 个并发子代理任务。
- 当并发子代理已达上限时，新的子代理任务排队等待；若等待超过 `30` 秒仍无空位，主代理放弃委派并自行处理该任务。回退执行时，主代理仅使用委派时已准备好的任务描述和上下文摘要作为输入，完成后仅将摘要结果纳入主上下文，以保持与子代理执行一致的上下文隔离效果。
- 主代理向子代理传递任务描述和必要上下文摘要，不传递完整会话历史；若子代理需要引用特定消息内容，由主代理在委派时显式提供。
- 子代理单次执行超时时间默认为 `120` 秒，该时限为子代理从启动到结束的总挂钟时间，子代理内部的所有操作（模型调用、MCP 工具调用、Skill 加载等）均计入此总时限；单次 MCP 工具调用仍受自身 `60` 秒超时约束，但两者独立判定——任一超时即视为触发。超时后子代理任务自动终止，主代理将该子任务标记为超时失败。
- 子代理执行失败（超时、模型错误、工具异常）时，主代理不自动重试，而是在回复中向用户说明该子任务未完成及失败原因，由用户决定是否重新发起。

### 7.6 Skills 规则

- Skills 目录位于应用根目录下的 `skills/` 文件夹，每个 Skill 为一个独立子目录。
- 每个 Skill 目录至少包含一个 `SKILL.md` 文件，该文件使用 YAML front matter 声明元信息，正文部分为 Skill 的指令内容。`SKILL.md` 的结构如下：
  - Front matter 必填字段：`name`（Skill 名称）、`description`（功能简述）、`match`（适用场景关键词或模式列表）。
  - 正文：提供给代理的知识、流程说明或执行约束，采用 Markdown 格式。
- Skill 目录内可包含附属资源文件（如模板、示例、脚本），由 `SKILL.md` 正文中引用。
- Skills 的作用是提供额外知识、流程和执行约束，不直接替代 MCP。
- 应用启动时扫描 `skills/` 目录，解析全部 `SKILL.md` 的 front matter，加载 Skill 元信息（名称、描述、适用场景）。
- 是否使用某个 Skill 由当前执行代理（主代理或子代理）根据任务内容与 Skill 的 `match` 字段自主匹配判断，不要求用户手动触发。

### 7.7 MCP 规则

- MCP 是统一工具接入层。
- 主代理和子代理都可以调用 MCP 工具。
- MCP 工具调用默认自动执行，不要求逐次确认。
- 是否调用 MCP 由当前执行代理自主判断。
- MCP Server 配置需要覆盖主流 transport：本地 `stdio`、远程 `streamable HTTP`，以及兼容历史生态的 `HTTP+SSE`。

### 7.8 浏览器兼容规则

- 目标浏览器优先支持桌面版 Chrome 和 Edge。
- 其他现代浏览器按 best effort 兼容。
- 若浏览器剪贴板未提供文件项，则粘贴行为按普通文本处理。

### 7.9 错误处理规则

- **模型请求超时或失败：** 该轮回复标记为失败状态，前端展示错误提示并允许用户重试该轮。
- **MCP Server 不可达：** 将该工具调用标记为失败，代理根据情况决定是否跳过该工具继续回答，并在回复中说明限制。
- **MCP 工具调用超时：** 单次 MCP 工具调用默认超时时间为 `60` 秒；超时后标记为失败，代理可选择跳过或使用替代方案继续回答。
- **Skill 加载失败：** 启动时若某个 Skill 的 `SKILL.md` 解析失败或目录结构不合法，跳过该 Skill 并在启动日志中记录警告，不影响其他 Skill 和系统正常运行。
- **附件文件丢失：** 消息中的附件卡片保留但标记为不可用，不影响其他消息和会话功能。
- **JSON 文件损坏：** 读取时自动尝试从 `.bak` 文件恢复；若恢复成功则正常加载；若 `.bak` 也不可用，则将该会话标记为不可读并在会话列表中提示异常，不影响其他会话。
- **前后端连接中断：** 前端在 SSE 连接断开或请求超时时，将当前轮回复标记为中断状态并展示连接异常提示，用户可手动重试该轮。
- **服务端异常终止：** 服务端在运行中崩溃或重启时，已持久化的用户消息保留，未持久化的助手回复丢失；重启后会话恢复至最后一次成功持久化的状态。
- **会话已有活动 `run`：** 后端拒绝新的请求并返回冲突状态，前端提示用户该会话正在生成中。

## 8. 存储要求

### 8.1 存储原则

- 不使用数据库。
- 使用 JSON 文件保存结构化数据。
- 使用本地目录保存附件原文件。
- 建议每个会话一个 JSON 文件，避免单文件过大。
- 每个会话需要维护独立 `revision`，用于多标签页写入冲突检测。
- 用户消息在接收时立即持久化；助手回复在 `run` 完成或中断时持久化；`run` 过程中尚未持久化的运行时中间状态（如工具调用的临时结果、子代理未完成的中间输出）不写入持久化消息历史；但用户触发停止后，模型已输出的文本内容视为已产生的回复片段，随 `interrupted` 状态一起持久化。
- JSON 文件写入采用原子写入模式：先写临时文件，确保写入完成后再 rename 替换目标文件，同时保留一份 `.bak` 副本用于异常恢复。

### 8.2 最小存储对象

- 全局设置
- 会话索引
- 单会话消息记录
- MCP Server 配置
- Skills 目录内容
- 附件原始文件

## 9. 交互要求

### 9.1 输入区

- 支持文本输入。
- 支持上传文件、拖拽文件、粘贴文件。
- 支持发送前附件预览与删除。
- 支持发送按钮与停止按钮。
- 默认 `Enter` 发送消息，`Shift+Enter` 换行。
- 若当前标签页会话已过时，发送时自动从当前节点分叉为新会话并保留用户草稿。

### 9.2 消息区

- 用户消息与助手消息需要清晰区分。
- 图片附件显示缩略图。
- 非图片附件显示卡片。
- 工具调用过程显示执行状态。
- 子代理结果默认显示摘要，可展开详情。
- 被中断的回复需要有视觉标记提示未完成。
- 失败的回复需要展示错误信息并提供重试入口。

### 9.3 设置页

- 设置页至少支持 `baseURL`、`apiKey`、`model` 和 MCP Server 管理。
- 保存成功后立即生效或在下次请求时生效。
- 若配置缺失或明显错误，需要给出明确提示。

## 10. 非功能要求

- 优先保证本地可运行。
- 结构简单，便于持续迭代。
- 不引入数据库。
- 不要求 Docker。
- 默认部署目标是本机或受信任内网环境，不以直接公网暴露为目标。
- 当前版本不做鉴权；若未来需要直接公网暴露，必须先补充访问控制、密钥保护与自动执行能力隔离。
- 工具、MCP、子代理默认自动执行。
- 多标签页同步按 best effort 实现，不以实时协同编辑为目标。
- 服务端收到终止信号时，先终止所有活动 `run` 并将已产生的部分内容持久化为中断状态，再释放 MCP 连接和文件句柄后退出。
- 系统应记录关键操作日志（启动与关闭、会话创建与删除、模型请求与响应状态、MCP 连接与工具调用状态、错误与异常），日志输出到标准输出或本地日志文件。

## 11. 风险与限制

- 当前设置与集成信息默认面向单用户受控环境；在无鉴权前提下不应直接暴露到公网。
- 即使在受控环境中，部署者也需要谨慎配置可自动执行的工具、MCP 与子代理能力。
- 不同 OpenAI 兼容后端在视觉、工具、文件处理能力上差异较大，需采用 best effort 兼容策略。
- 浏览器对剪贴板中文件项的支持存在差异，需以前端能力检测和降级方案兜底。
- 无数据库方案在会话规模扩大后会面临文件组织与检索效率问题，但当前阶段可接受。

## 12. 验收标准

### 12.1 聊天验收

- 用户可以新建会话并正常发送消息。
- 助手回复支持流式输出。
- 用户可以停止当前轮生成，已输出内容保留并标记为中断。
- 历史会话可恢复并继续对话。
- 用户可以在不同标签页中同时对不同会话发起问答。
- 同一会话在另一个标签页已更新时，当前标签页发送会自动分叉为新会话并保留用户草稿。
- 同一会话在已有活动 `run` 时，其他标签页不能再次发起新的 `run`。
- 模型请求失败时展示错误提示并可重试。
- 用户可以清空所有会话，操作需二次确认，确认后全部会话及附件被删除。

### 12.2 附件验收

- 用户可以通过上传、拖拽、粘贴三种方式添加附件。
- 用户可以在发送前删除单个附件。
- 消息发出后，附件可在消息中正常展示。
- 图片附件显示缩略图，非图片附件显示卡片。
- 图片附件可点击预览大图。
- 图片附件在模型支持视觉时可参与理解；非图片附件在存在合适 MCP 工具时可先解析再参与后续处理。
- 当非图片附件缺少可用解析工具时，系统会明确提示限制，而不是假装已理解文件内容。
- 删除会话后，该会话目录下的附件文件被正确清理。

### 12.3 设置验收

- 用户可以保存并读取 `baseURL`、`apiKey`、`model` 与 MCP Server 配置。
- 当配置不完整时，系统阻止发送请求并提示原因。

### 12.4 能力验收

- 系统可以加载应用内 Skills。
- Skill 的选择由代理根据任务内容自主判断。
- 系统可以接入并调用 MCP Server 提供的工具。
- 系统可以接入主流 MCP transport：`stdio`、`streamable HTTP`、兼容 `HTTP+SSE`。
- 主代理可以将合适任务委派给子代理。
- 子代理可以在执行中自主调用工具、MCP、Skill，并向主代理返回摘要结果。
- 子代理不会递归委派，单 `run` 并发子代理不超过 3 个。
- 长会话在消息累计超过上下文窗口容量阈值时，系统自动对较早消息生成摘要并用于后续模型请求，完整历史仍可在前端查看。

### 12.5 存储验收

- 关闭并重新启动应用后，会话与设置可恢复。
- 附件原文件仍可访问。
- JSON 文件采用原子写入，异常中断不易造成文件整体损坏。
- JSON 文件损坏时可从 `.bak` 自动恢复。
